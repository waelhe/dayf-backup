## I. الثقة أولاً — Escrow & Disputes

منصة ضيف قائمة على الثقة المالية. كل قرار تقني يجب أن يحمي هذه الثقة.

**القواعد الثابتة:**
- Escrow مطلوب لكل حجز يتجاوز صفراً — لا استثناء
- Soft Delete إلزامي لكل البيانات المالية والمراجعات
- سجل Audit كامل لكل تغيير في الحجوزات والمدفوعات
- لا يمكن حذف مراجعة موثقة (Verified Review) أبداً — فقط الإخفاء مع التوثيق

---

## II. AI كطبقة تحسين — وليس أساساً

كل وظيفة في ضيف يجب أن تعمل بدون AI أولاً.

**القواعد الثابتة:**
- كل ميزة ذكاء اصطناعي لها fallback كلاسيكي يعمل
- AI provider قابل للتبديل (OpenRouter/Anthropic/Google) بدون تغيير الكود
- لا يُعرض خطأ AI للمستخدم — يُعرض النظام الكلاسيكي بدلاً
- استجابة AI المحرجة (offensive/incorrect) تُعاد للنظام الكلاسيكي تلقائياً
- سجّل كل استدعاء AI: التكلفة، المدة، النتيجة

---

## III. البيانات السياحية السورية مقدسة

المحتوى السياحي الموثوق هو الميزة التنافسية الأساسية لضيف.

**القواعد الثابتة:**
- كل Destination/Activity/Tour يحتاج موافقة Admin قبل النشر
- الصور لا تُحذف — تُؤرشف فقط
- اللغة العربية (RTL) هي اللغة الأولى في كل واجهة
- بيانات الأسعار السورية (ليرة + دولار) تُخزن دائماً بالاثنين
- لا يمكن لـ AI توليد بيانات Destination بدون مراجعة بشرية

---

## IV. المراجعات هي العمود الفقري

نظام المراجعات هو ما يربط الأنظمة الخمسة. يُبنى أولاً، يُحمى دائماً.

**القواعد الثابتة:**
- كل مراجعة تحتاج Booking مؤكد (Verified Purchase)
- المراجعات تُغذي التوصيات، ونظام السمعة، ونظام Gamification
- تغيير خوارزمية ترتيب المراجعات يحتاج موافقة موثقة في DECISIONS.md
- لا يمكن للمضيف حذف مراجعة — فقط الرد عليها

---

## V. الاستقلالية بين الوحدات عبر الأحداث فقط

**القواعد الثابتة:**
- لا استدعاء مباشر بين modules — Event Bus فقط
- `import { BookingService } from '../bookings'` → ممنوع
- `eventBus.publish('booking.confirmed', data)` → مسموح
- كل وحدة تُعرّف واجهتها عبر `index.ts` فقط

---

## VI. الأمان بالافتراض — Security by Default

**القواعد الثابتة:**
- Deny by Default: لا صلاحية إلا إذا مُنحت صراحةً
- كل API route يمر بـ auth middleware — لا استثناء
- Zod validation على كل input خارجي
- لا secrets في الكود — `.env` فقط
- Rate limiting على كل endpoint عام
- httpOnly Cookie للـ tokens — لا localStorage أبداً

# 📋 سجل القرارات المعمارية

### ADR-001: نظام Escrow كطبقة ثقة مالية
**الحالة:** ✅ مُطبق
**القرار:** Escrow مدمج بدلاً من الدفع المباشر

### ADR-002: SQLite في مرحلة MVP
**الحالة:** 🟠 قيد التغيير — الهجرة إلى Supabase
**القرار:** SQLite + Prisma للـ MVP
**التالي:** PostgreSQL + Supabase

### ADR-003: الأنظمة الخمسة كوحدات مستقلة
**الحالة:** ✅ مُقرر
**القرار:** كل نظام = module مستقل يتواصل عبر Event Bus

### ADR-004: نظام المراجعة أولاً
**الحالة:** 🟠 مُقرر
**القرار:** نظام المراجعة هو التالي

### ADR-005: AI عبر z-ai-web-dev-sdk
**الحالة:** ✅ مُطبق
**القرار:** كل AI يمر عبر SDK

### ADR-006: Socket.io للـ Real-time
**الحالة:** 🟡 مؤجل
**القرار:** Socket.io للـ chat و notifications
